# NekoOAuth2Server ‚Äî Roadmap / TODO

This is the working plan for building NekoOAuth2Server: a self-hosted OIDC/OAuth 2.1 identity provider ‚Äî Logto-inspired, but self-built ‚Äî meant to be the *one* login server every Neko\* project federates through, instead of each project running its own separate Discord OAuth app and disconnected user table.

Legend: `[ ]` not started ¬∑ `[~]` in progress ¬∑ `[x]` done

---

## Phase 0 ‚Äî Planning & Decisions

- [x] Write initial README.md, TODO.md
- [x] Decide project license ‚Üí **AGPL-3.0**. This server holds auth secrets and user data for every other Neko\* project; the network-use clause means anyone who forks this and runs a modified version as a hosted service has to release their changes too ‚Äî closes the "quietly resell my work" loophole plain GPL leaves open for server software.
- [x] Decide core language/runtime ‚Üí **Node.js + TypeScript**, confirmed after auditing every other Neko\* web project. Node/TS is used in 14 of 15 surveyed projects; it's the only realistic choice for staying consistent with the rest of the ecosystem this server has to interoperate with.
- [x] Decide database ‚Üí **PostgreSQL** (not the more common MySQL/MariaDB seen across older Neko\* projects) ‚Äî better fit for the JSON columns, row-level constraints, and relational RBAC/multi-tenancy data this project actually needs, and it's what Logto itself uses.
- [x] Decide repo layout ‚Üí **pnpm monorepo**: `apps/server` (OIDC core), `apps/console` (admin dashboard, added when Phase 8 starts), `packages/shared` (types/schemas), `packages/sdk` (client library for downstream Neko\* apps to consume) ‚Äî same shape as `account-recovery-keychain`.
- [x] Decide how to build the OIDC/OAuth core ‚Üí **do not hand-roll authorization-code/PKCE/JWT crypto from scratch.** Build on a mature, widely-audited Node OIDC provider library (leading candidate: `panva/oidc-provider`, which is also what Logto's own `core` package is built on) and put the "own version" effort into the data model, multi-tenancy, RBAC, admin console, and downstream SDK instead. A bug in hand-rolled token-signing code would compromise every federated Neko\* site at once.
- [x] Researched Logto's actual architecture (`logto-io/logto` on GitHub) for a rough reference map ‚Äî see notes below. Confirms the plan rather than changing it.
- [x] Decide the connector architecture ‚Üí **generic, config-driven connector types, not bespoke code per provider.** Logto's own `connectors` packages confirm this pattern: a standard connector interface (`getAuthorizationUri`, `authorizationCallbackHandler`, `getAccessToken`, `getUserInfo`) is implemented once each for a **generic OAuth 2.0 connector** and a **generic OIDC connector**; individual providers (Google, Discord, etc.) are then just pre-filled *config profiles* (authorize/token/userinfo URLs, client id/secret, scopes) of one of those two generic types. Only providers with genuinely non-standard flows (SAML, or ‚Äî in our case, VRChat, which has no OAuth2 at all) need real custom code. This is the same "reuse a proven shape, don't hand-roll 40 times" reasoning as the OIDC-library decision above.
- [x] Decide ORM ‚Üí **Prisma** (matches `account-recovery-keychain`, our newest/most security-conscious project). Note for reference, not a reason to switch: Logto's own core uses Slonik (a query builder) + Zod schemas instead of a traditional ORM ‚Äî a valid alternative approach, but Prisma keeps us consistent with our own house pattern.
- [x] Decide account-deletion / data-lifecycle model ‚Üí a **signed webhook system** (HMAC-signed payloads, delivery log, matching Logto's Console ‚Üí Webhooks model) so that when a user deletes their account here, every downstream Neko\* app subscribed via the SDK gets a `user.deleted` event and purges its own locally cached copy of that user's data. Deleting an account on this server must actually remove the person's data everywhere it's been shared, not just here.
- [x] Confirm the OIDC provider library ‚Üí **`panva/oidc-provider`**, confirmed rather than assumed. As of this check: v9.8.6 (actively released, 2,848 commits), MIT licensed, ~3.8k GitHub stars, and genuinely **OpenID Certified** (Basic, Implicit, Hybrid, FAPI 1.0, FAPI CIBA, FAPI 2.0 profiles) ‚Äî not just popular, independently certified spec-conformant. It implements RFC 7636 PKCE natively and documents a clear custom-`Adapter` interface (`upsert`/`find`/`findByUserCode`/`findByUid`/`consume`/`destroy`/`revokeByGrantId`/`findByPayloadValueSet`), which is exactly the seam Phase 1 needs to back it with our own Prisma/Postgres storage instead of its default in-memory adapter. Also confirmed to be what Logto's own `core` package is built on ‚Äî same library, independently arrived at.
- [x] Decide the shape of the "shared identity" data model:
  - **`User`** (this server's own identity) ‚Äî `id`, `primaryEmail` (nullable + `emailVerified`), `displayName`, `avatarUrl`, `status` (`active`/`suspended`/`deleted`), timestamps. This is the *only* place a person's core identity lives ‚Äî deliberately thin.
  - **`LinkedIdentity`** ‚Äî one row per external platform link: `userId` FK, `provider` (`discord`/`vrchat`/`roblox`/`twitch`/`vpzone`/‚Ä¶), `providerUserId`, `providerUsername`, `verifiedVia` (`oauth` for standard connectors, `bio` or `friend_request` for VRChat's bot-verified flow ‚Äî see Phase 4), `linkedAt`. Unique on `(provider, providerUserId)` so one external account can't be linked to two different NekoOAuth2Server users. This is Phase 5's multi-connector linking, generalized from `SocialLinkUpOnly`'s single-project `LinkedAccount` table.
  - **`ClientConsent`** ‚Äî `userId` + `clientId` FKs, `grantedScopes` (string array), `grantedAt`, `revokedAt` (nullable). This is *the* answer to "what can a downstream app read": a client only ever receives the `User` fields and `LinkedIdentity` rows covered by scopes it was actually granted ‚Äî e.g. `profile`, `email`, and per-platform scopes like `identity:discord` / `identity:vrchat` ‚Äî nothing else, and it doubles as the self-service "apps with access to your data" screen (Phase 5/8).
  - **Explicitly not stored here**: any per-project data (a game inventory, a site-specific subscription tier, etc.). That stays in each downstream Neko\* app's own database, keyed by the `sub` claim this server issues. This server is identity + linked platforms + consent, nothing more ‚Äî keeps the single point of failure as small as it can be.

### Reference notes from studying Logto's architecture (not our code, just what informed the decisions above)

- Monorepo shape: `core` (OIDC server), `console` (admin SPA), `experience` (end-user sign-in/sign-up UI, kept separate from the admin console), `schemas` (DB row shapes via Zod), `connectors` (one package per provider/protocol), `toolkit` (shared libs incl. `connector-kit`, the interface connectors implement), `cli` (migrations/seeding).
- Multi-tenancy: **organization-level** multi-tenancy (tenants-within-a-product, RBAC per organization) is in Logto's open-source core. *Console-level* multi-tenancy (managing several separate Logto instances from one admin login) is Cloud-only ‚Äî not relevant to us since we're self-hosting one instance for ourselves.
- RBAC: global app-level roles/permissions plus an "organization template" ‚Äî one shared role/permission set applied consistently per-organization, with a user able to hold different roles in different organizations. Reasonable shape to borrow for Phase 3.
- Email: pluggable "email connectors" (built-in/SendGrid/Mailgun/SMTP/generic HTTP), each implementing a common send interface; templates keyed by `usageType` (`Register`, `SignIn`, `ForgotPassword`, `Generic`, etc.) with a `{{code}}` placeholder. This is exactly the model Phase 6 below follows, scoped to SMTP only for v1.

## Phase 1 ‚Äî Core Server Skeleton

- [x] Scaffold `apps/server` ‚Äî minimal Express + TypeScript app with a `/health` endpoint, env-based config, and a build/dev script. No OIDC logic yet ‚Äî just proves the workspace boots. **Verified live**: `pnpm install` succeeds, `pnpm --filter server dev` boots, and `curl localhost:4000/health` returns `{"status":"ok"}`.
- [x] `docker-compose.dev.yml` (repo root) for **local dev only** ‚Äî Postgres 16 + adminer, matching the credentials in `.env.example` ‚Äî separate from the full production Docker setup in Phase 10.
- [x] Prisma schema (`apps/server/prisma/schema.prisma`): `Tenant`, `Client`, `User`, `LinkedIdentity`, `ClientConsent` per Phase 0's data model, plus `OidcModel` ‚Äî a generic key/value table backing every oidc-provider model *except* Client (Session, AccessToken, AuthorizationCode, RefreshToken, Grant, Interaction, etc. all share it, namespaced by `type`), the standard pattern for SQL-backed oidc-provider adapters rather than one bespoke table per model. There's no separate `Session` table ‚Äî oidc-provider's "Session" model is one more `type` row in `OidcModel`, not a hand-rolled concept of our own.
  - One real correction made along the way: the original plan said `clientSecretHash`. Renamed to plain `clientSecret` ‚Äî oidc-provider's built-in `client_secret_basic`/`client_secret_post` verification does a constant-time comparison against the *raw* configured secret, so a one-way hash would silently break auth. Noted in the schema and flagged for Phase 9 to revisit as reversible envelope encryption at rest instead.
  - Migration SQL generated and committed at `apps/server/prisma/migrations/20260101000000_init/`.
- [x] Wired `oidc-provider` (`panva/oidc-provider`, per Phase 0) into `apps/server` (`src/oidc/provider.ts`), with a custom adapter (`src/oidc/adapter.ts`) ‚Äî `oidcModelAdapter.ts` implements the generic `OidcModel`-table adapter (upsert/find/findByUid/findByUserCode/consume/destroy/revokeByGrantId, mirroring oidc-provider's own memory adapter), and `clientAdapter.ts` overrides just the `Client` model to read from our own `Client` table instead, so the admin console (Phase 8) will CRUD real columns rather than JSON blobs.
- [x] PKCE required for every client type, including confidential ones ‚Äî `pkce: { required: () => true }` (a top-level `Configuration` key, not a `features` flag ‚Äî the installed library's own source was checked directly after `@types/oidc-provider@9.5.0` turned out not to type it). **Verified with a real, passing 7-test suite** (`src/oidc/pkce.test.ts`, supertest against `oidcProvider.callback()`), not just configured and assumed correct:
  - a public client's `/auth` request with no `code_challenge` is rejected (`invalid_request`, "policy requires PKCE")
  - a **confidential** client's `/auth` request with no `code_challenge` is *also* rejected ‚Äî the actual point of this policy, since RFC 9700 alone wouldn't require it for confidential clients
  - a request that *does* supply a valid `code_challenge` proceeds normally (positive control ‚Äî proves the check is specifically about PKCE, not blocking everything)
  - a code exchange with the wrong `code_verifier` is rejected (`invalid_grant`)
  - a code exchange with no `code_verifier` at all is rejected (`invalid_grant`)
  - a code exchange with the **correct** `code_verifier` succeeds and returns a real `access_token` ‚Äî proving the mechanism actually works end-to-end, not just that it always errors
  - the confidential-client code path is independently re-checked at the token endpoint too
  - One real bug the test suite caught in itself: the first "wrong code_verifier" attempt used a hand-typed 42-character string, one short of RFC 7636's 43-character minimum, so oidc-provider correctly rejected it as a format error (`invalid_request`) rather than reaching the semantic mismatch check (`invalid_grant`) the test meant to exercise. Fixed by generating the wrong verifier the same way as a real one (`crypto.randomBytes(32).toString("base64url")`), just discarded rather than used for the challenge.

**A note on how this was verified, in the interest of honesty**: this sandbox has no Docker and no local Postgres install. Schema and query-level verification (does the schema apply, do the adapter's queries round-trip correctly, does the PKCE policy actually behave as configured) was done against [`@electric-sql/pglite`](https://github.com/electric-sql/pglite) ‚Äî a real Postgres engine compiled to WASM, wire-compatible enough for `prisma db push` and the full application/test stack ‚Äî proxied over TCP via `@electric-sql/pglite-socket` so the standard `postgresql://` connection string worked unmodified. Neither package is a dependency of the project itself; they only ever ran in a scratch directory outside the repo. `prisma migrate dev` itself doesn't work against it (it needs a shadow database via `CREATE DATABASE`, which this single-logical-database engine doesn't support), so the committed migration SQL was instead generated with `prisma migrate diff --from-empty` ‚Äî a schema-only diff that needs no live database ‚Äî and its correctness was separately confirmed by watching `db push` apply that identical schema cleanly. The one thing **not** re-verified here is the real `docker-compose.dev.yml` path itself (no Docker daemon available) ‚Äî worth a quick real run before this is treated as fully proven on a normal dev machine.

**Update from Phase 2**: this substitute has a real limit worth knowing about ‚Äî under the heavier, multi-hop HTTP traffic of `e2e.test.ts`'s 9 tests, it reliably degrades enough that whichever test file runs *after* it in the same `vitest` process fails to connect at all. Both `pkce.test.ts` and `e2e.test.ts` pass 100% individually against a freshly restarted instance; it's specifically the combined-in-one-process, heavy-load case that's unreliable. This reads as a connection-handling limit of the pglite socket multiplexer, not a product bug ‚Äî but it's also a real signal that this substitute has reached the edge of what it's good for. Real Postgres via `docker-compose.dev.yml` is worth switching to for any further phases' verification.

## Phase 2 ‚Äî OAuth 2.1 / OIDC Compliance

- [x] **Authorization code flow, end to end, against a real test client** ‚Äî driven over real HTTP (`src/oidc/e2e.test.ts`), not internal-model shortcuts: `/oidc/auth` ‚Üí our own login form ‚Üí our own consent form ‚Üí redirect back to the client with a code ‚Üí `/oidc/token` exchange ‚Üí real `access_token`/`id_token`. This required building a real (if deliberately unstyled) interaction UI, since `oidc-provider`'s built-in `devInteractions` is explicitly "not for production" ‚Äî see the new `src/oidc/interactions.ts` and `src/oidc/provider.ts`'s `findAccount` (backed by Prisma, needs a real `User.passwordHash` ‚Äî added to the schema, see Phase 0's note on this being a placeholder baseline credential, not a final decision on end-user auth). **This exposed a real gap in the roadmap**: there was no phase covering the end-user sign-in/consent UI at all (Phase 8 only covers the *admin* console). The interaction routes built here are functional but intentionally minimal/unstyled; a real sign-in experience is still open ‚Äî see Open Questions.
- [x] **Refresh token rotation with reuse detection** ‚Äî no custom config needed: `oidc-provider`'s default `rotateRefreshToken` policy already rotates every refresh token for public clients unconditionally. **Verified with a real 3-step replay test**: use refresh token #1 ‚Üí get a new refresh token #2 (rotation confirmed by the two values differing) ‚Üí replay the now-consumed #1 ‚Üí rejected (`invalid_grant`) ‚Üí then try the legitimately-rotated #2 ‚Üí *also* rejected. That last step is the one that actually proves "revokes the whole token family," not just "the specific replayed token is dead."
- [x] **Client credentials flow for service-to-service calls** ‚Äî `features.clientCredentials.enabled: true`, plus one real resource indicator (`INTERNAL_API_RESOURCE`, a placeholder identifier standing in for "some future Neko\* backend API") wired through `features.resourceIndicators.getResourceServerInfo`. Verified both with and without an explicit `resource` parameter ‚Äî a client_credentials token doesn't require one at all if the caller just wants an unscoped-to-any-API token.
- [x] **JWKS endpoint + key rotation strategy** ‚Äî `scripts/generate-jwks.ts` generates a real persistent RS256 keypair (set as the `JWKS` env var); without it the server falls back to an ephemeral dev keystore that changes every restart (fine for local dev, breaks anything relying on a stable key in real use). **Rotation strategy is manual, not automated**: add a new key alongside the old one in the `keys` array (never replace outright) so tokens signed with the old key keep verifying until every token issued under it has expired, then remove the old key. No scheduler exists to do this automatically yet ‚Äî that's real future work, not implied by what's here.
- [x] **Discovery document** ‚Äî already live since Phase 1; this phase specifically verified completeness (`grant_types_supported`, `code_challenge_methods_supported`, `scopes_supported`, `claims_supported`, etc.) rather than just that the endpoint responds. **One real, non-obvious behavior found while testing this**: `authorization_endpoint`/`token_endpoint`/`jwks_uri` are host-qualified using the *serving request's* Host header, not statically fixed to the configured issuer string ‚Äî correct behavior for reverse-proxy-friendly deployments, but it means naive tests (or naive downstream clients) asserting an exact static URL for these will be wrong; only `issuer` itself is the static, configured value.
- [x] **Scope/claims model** ‚Äî `openid`/`profile`/`email` claims configured, plus two resource-specific scopes (`internal:read`/`internal:write`) for the client-credentials resource above. **Real, confirmed distinction**: profile/email claims are delivered via the `/oidc/me` userinfo endpoint when requested with an access token, *not* embedded in the code-flow `id_token` (which only carries `sub` by default) ‚Äî verified by actually calling `/oidc/me` and checking claims appear/disappear based on granted scope, not by inspecting the id_token (an earlier version of this test checked the wrong place and looked like it passed for the wrong reason).

**Bugs this phase's real testing caught (all fixed, not glossed over):**
- Setting a custom `scopes: [...]` config array to register `internal:read`/`internal:write` **replaced**, rather than extended, `oidc-provider`'s default `['openid', 'offline_access']` ‚Äî which silently disabled the `refresh_token` grant entirely (it's only enabled when `'offline_access'` is present in that exact list) and broke every client whose registered `grant_types` included `refresh_token`. Caught by the client-metadata-validation error itself (`grant_types can only contain 'implicit', 'authorization_code', 'client_credentials'`), not by guessing ‚Äî fixed by including the defaults explicitly alongside the new scopes.
- The planned `User.clientSecretHash`-style thinking almost repeated itself here: `Client.scope` (a client's *allowed-to-request* scopes) and a resource server's *own* granted scopes (`getResourceServerInfo`'s return value) are two separate validation layers ‚Äî a scope has to be recognized in both places, not just one.
- `pkce.test.ts` (Phase 1) started failing after `findAccount` was wired up for real: it used a fabricated `accountId` string that was never a real `User` row, which worked when `findAccount` was still `oidc-provider`'s default passthrough (accepts any id) but broke once a real Prisma-backed lookup replaced it. Fixed by seeding a real test `User` and referencing its actual id ‚Äî arguably a more correct test than before, not just a patch.

## Phase 3 ‚Äî Multi-Tenancy & RBAC

- [x] **Tenant isolation model, decided**: a single shared `Tenant` row (`neko`) for all first-party Neko\* apps, not one per project. Real isolation lives at the **Client** level (each downstream app is its own `Client` row with its own `Role`s), not the tenant level ‚Äî matches Phase 0's framing that multi-tenancy shouldn't be a requirement for "a handful of trusted first-party apps." The `Tenant` model stays in the schema unused-for-isolation for now, so nothing blocks adding real multi-tenant SaaS behavior later if this server is ever offered to third parties.
- [x] **Role/permission schema**: `Role` (belongs to exactly one `Client`, has a plain `permissions String[]` rather than a separate `Permission` table ‚Äî simple enough not to need the extra join) and `UserRole` (many-to-many, `User` ‚Üî `Role`). **Deliberately per-client, not per-tenant**: with a single shared tenant, tenant-scoped roles wouldn't add any real isolation ‚Äî two rows named "admin" on different clients are unrelated by design, so a name collision across projects can never leak permissions.
- [x] **RBAC in issued token claims**: a new `roles` scope (`src/oidc/provider.ts`'s `claims.roles`) that, when granted, adds `roles`/`permissions` arrays to `findAccount`'s claims ‚Äî resolved fresh from the database, scoped to `ctx.oidc.client.clientId` (the project being authenticated to *right now*), not every role the user holds anywhere. This is what makes cross-project isolation automatic instead of something every caller has to remember to filter for itself.
- [x] **RBAC enforcement on an API**: there's no real admin API yet (Phase 8 builds it), so this phase built the enforcement *mechanism* ‚Äî `src/rbac/requirePermission.ts`, an Express middleware ‚Äî and a small example protected route (`GET /api/internal/admin-ping`) demonstrating the exact pattern Phase 8's real admin API should follow. Deliberately re-checks roles/permissions against the current database on every call rather than trusting a claim embedded in a token, so revoking a role takes effect immediately rather than only once the access token expires.
- [x] **Tests: cross-project isolation, proven, not assumed** (`src/rbac/crossProjectIsolation.test.ts`, 4 tests, all passing) ‚Äî one user, one "admin" role with an `admin:access` permission granted **only** on Project A: authenticating to Project A shows the role in `/oidc/me` and passes the protected endpoint; authenticating to the *same account* via Project B shows zero roles and gets a 403 from the identical protected endpoint. Proves the isolation is real, not just that the schema has a `clientId` column.

Refactored `e2e.test.ts`'s HTTP-driven authorize/login/consent flow into a shared `src/testSupport/httpAuthFlow.ts` helper so this phase's RBAC test (and any future one) doesn't reimplement it ‚Äî confirmed no regression by rerunning Phase 1/2's suites after the extraction.

## Phase 4 ‚Äî SSO / Upstream Connectors

**Architecture, built as planned**: two generic, config-driven connector types (see Phase 0 decision) implementing one shared interface ‚Äî `src/connectors/types.ts`'s `UpstreamConnector` (`getAuthorizationUri`, `exchangeCode`, `getUserInfo`) ‚Äî plus VRChat's genuinely custom bot-verified module, which does not implement this interface at all (it has no authorization redirect).

**Scope decided for this pass** (see conversation): build the generic engine plus the four providers with concrete, confirmed endpoint details (Discord/Roblox/Twitch/VPZone). The 13-provider backlog (Google, GitHub, Microsoft, etc.) is left unstarted ‚Äî each needs its own real endpoint verification against current provider docs, not something to bulk-guess through.

- [x] **Generic OAuth 2.0 connector** (`src/connectors/oauth2Connector.ts`) ‚Äî config-driven (authorization/token/userinfo URLs, client id/secret, scope), tri-state `pkce: "required" | "optional" | "unsupported"` exactly as planned (not a single global assumption), pluggable `userInfoHeaders` (needed for Twitch's `Client-Id` requirement) and `tokenAuthMethod` (`client_secret_post` vs `client_secret_basic`). **Verified with 6 tests** against a real local mock HTTP server (not just unit-testing in isolation) ‚Äî correct authorization URL construction with/without PKCE, correct S256 challenge derivation, correct token-exchange request body/auth header for both client-auth methods, correct userinfo parsing.
- [x] **Generic OIDC connector** (`src/connectors/oidcConnector.ts`) ‚Äî fetches the provider's real discovery document instead of hardcoding endpoints, validates the id_token's signature/issuer/audience via `jose` against the provider's real JWKS before trusting anything in it. **Verified against our own real, live oidc-provider** (self-referential ‚Äî genuinely more thorough than a mock, since it's a real spec-compliant OIDC server already proven in Phases 1-3): a real discovery fetch, a real code exchange, a real signed id_token that verifies, and a forged-token-signed-by-an-unrelated-key rejection (proving the signature check is real, not a no-op).
  - **Real bug this test caught**: the connector originally validated the id_token's `iss` against the URL passed in to *locate* the discovery document. Per spec, the correct check is against the discovery document's own `issuer` field, which can legitimately differ (confirmed live: our own server's discovery endpoints are host-derived per-request, but `issuer` itself stays the static configured value). Fixed to read `discovery.issuer`, matching what a compliant OIDC client is actually supposed to check.
- [x] Connector plugin interface documented in `src/connectors/types.ts` directly (JSDoc), not a separate doc file ‚Äî the interface is small enough that the code comment is the documentation.

### Wired into the actual login flow, not just built and left unused

- [x] The interaction login page (`src/oidc/interactions.ts`) now lists a "Sign in with X" link per enabled connector (a provider only appears if its `_CLIENT_ID`/`_CLIENT_SECRET` env vars are actually set ‚Äî booting without Discord configured just means Discord login isn't offered, not a crash).
- [x] `GET /oidc/interaction/:uid/upstream/:providerId` (start) and `.../callback` routes: the OAuth `state` parameter carries an HMAC-signed `{ uid, provider, codeVerifier }` payload (`src/connectors/state.ts`) rather than needing a separate table to persist it across the redirect-out/redirect-back round trip.

- [x] **Verified end to end with a real local mock provider standing in for a real one** (`src/connectors/upstreamLogin.test.ts`, 2 tests): the full real-HTTP round trip (login page ‚Üí start ‚Üí real redirect to the mock provider ‚Üí callback ‚Üí token exchange ‚Üí userinfo ‚Üí account creation) completes and reaches the client's redirect_uri with a code; a second login via the same upstream identity reuses the same `User` row rather than creating a duplicate.

## Phase 5 ‚Äî Shared Identity, Account Linking & Data Lifecycle

- [x] **Global vs. per-project profile fields, confirmed** ‚Äî no schema change needed; this was effectively already decided in Phase 0 and this phase's build only reinforced it. Global (lives on `User`, readable via scope-gated `/oidc/me` and now writable via `PATCH /api/profile`): `primaryEmail`/`emailVerified`, `displayName`, `avatarUrl`. Everything else (a game inventory, a site-specific subscription tier, per-project roles/permissions) stays out of this server entirely, keyed by the `sub` claim in each downstream app's own database.
- [x] **Consent screen shows plain language, not raw scope identifiers** ‚Äî `src/account/scopeDescriptions.ts` maps each scope to a human sentence (e.g. `profile` ‚Üí "Your name and profile picture"); `interactions.ts`'s consent page renders that list instead of a raw JSON dump of `missingOIDCScope`/`missingResourceScopes`. Falls back to `Access to "<scope>"` for anything unlisted rather than hiding an unrecognized scope.
- [x] **Multi-connector linking, built as a real self-service account portal** ‚Äî new `src/account/router.ts`, mounted at `/account`, with its own signed-cookie session (`src/account/session.ts`) deliberately separate from oidc-provider's internal interaction session (this is a person managing their account directly, not an OAuth client's login flow). Login, view profile + linked identities, link a new upstream connector (reusing Phase 4's `UpstreamConnector` interface in a "link to my existing account" mode via a `mode: "link"` discriminant added to the shared signed-state helper, `connectors/state.ts`), and unlink. **A linked identity already claimed by a *different* account is refused (409)**, not silently reassigned ‚Äî verified with a real test, not just a schema constraint assumption.
- [x] **API for a downstream app to read/update shared profile fields** ‚Äî reading was already live since Phase 2 (`/oidc/me`, scope-filtered); this phase added the write side, `PATCH /api/profile` (`src/account/profileApi.ts`), gated per-field by the access token's own granted scope (a `profile`-only token can change `name`/`picture` but a 403 on `email`, and vice versa) ‚Äî checked against the *token's* scope, not just "logged in," so a narrowly-scoped token can't creep into fields it was never granted.
- [x] **Self-service account deletion** ‚Äî `POST /account/delete` (`src/account/deleteAccount.ts`). Also revokes every active `Grant` (and the tokens/codes issued under it) and any live `Session` for that account ‚Äî deleting the `User` row alone would have left existing access/refresh tokens and browser sessions valid until they naturally expired, which isn't really "deleted."
- [x] **Deletion cascade via signed webhook** ‚Äî `WebhookEndpoint`/`WebhookDelivery` Prisma models, `src/webhooks/deliver.ts`. Notifies every client the user has a `ClientConsent` relationship with (the structured signal for "this client has definitely authenticated this user" ‚Äî `LinkedIdentity` is about upstream providers *we* consume, not which of *our* clients know this user) with an HMAC-signed `X-Neko-Signature` header over a `{ event: "user.deleted", data: { sub } }` payload, and logs the outcome (status code or error) to `WebhookDelivery`. **Verified with a real local HTTP receiver**: correct signature (independently recomputed and compared in the test), correct payload, and a real delivery log row.
- [x] **Migration plan for existing per-project user tables** ‚Äî documented below, not just "figure it out later":
  1. For each existing Neko\* project's user table (typically keyed by a platform ID like a Discord snowflake), create one `LinkedIdentity` row per platform account already on file (`provider` = the platform, `providerUserId` = the existing platform ID, `verifiedVia: "oauth"` for anything that was already a real OAuth login).
  2. If two of a project's existing users happen to share an email already present in `User.primaryEmail` from an earlier-migrated project, they collapse into the same shared `User` ‚Äî intentional, this *is* the point of a shared identity, but back up first since it's a one-way merge.
  3. Everything project-specific in that old user table (a subscription tier, an in-game inventory, moderation notes) stays in that project's own database, now keyed by the new shared `User.id` (the token's `sub`) instead of the platform ID directly ‚Äî a one-time re-key, not a data migration into this server.
  4. Register the project as a `Client` here, switch its login flow to redirect through this server (Phase 7's SDK) instead of calling Discord/etc. directly, and register a `WebhookEndpoint` so it gets deletion notifications going forward.

**One item added beyond the original checklist, worth calling out**: wired Phase 4's previously-unwired VRChat bot module into this account portal (`POST /account/link/vrchat/bio` and `/friend`), fulfilling the two items that phase left open pending Phase 5's linking UI. Verification runs in the background (fire-and-forget from the request handler ‚Äî bio/friend verification can take up to 5 real minutes of polling, far longer than any HTTP request should stay open) and the resulting `LinkedIdentity` just appears on `/account` once it completes.

**Bugs/gaps this phase's tests caught:**
- `verification.ts`'s poll-interval/timeout env-var overrides (added so integration tests don't wait real minutes) were originally read into top-level `const`s ‚Äî which silently locked in "unset" whenever this module happened to be reached before `config.ts`'s `import "dotenv/config"` elsewhere in the module graph (import-order dependent, not something visible from reading either file in isolation). Caught by the VRChat linking test timing out with the override seemingly ignored; fixed by reading `process.env` lazily inside the function instead of at module-load time.
- My own test seed for the profile-API test registered a public client (`isConfidential: false`, no secret) but forgot to set `tokenEndpointAuthMethod: "none"`, so it defaulted to `client_secret_basic` and the authorize request failed with `client_secret is mandatory property`. A test-authoring mistake, not a product bug ‚Äî but a real one, caught by actually running it rather than assuming the seed was right.
- An early version of the "unlink" test asserted the provider's id disappeared from the account page *entirely* ‚Äî wrong, since an unlinked provider correctly reappears in the "linkable" list. Fixed the assertion to check the specific linked-item text instead of a blanket page-content check.

## Phase 7 ‚Äî Client SDK for Downstream Neko\* Apps

- [x] **`packages/sdk`** (`@nekosunevr/oauth2-sdk`) ‚Äî `NekoAuthClient`/`createNekoAuthClient`: PKCE generation (mandatory, no tri-state needed here ‚Äî unlike the server's *upstream* connectors, our own server always requires it), OIDC discovery, authorize-URL construction, code exchange, refresh, `verifyIdToken` (signature/issuer/audience against the real JWKS via `jose`, exactly mirroring the fix already made in the server's own `oidcConnector.ts`), and scope-gated userinfo fetch.

- [x] **Webhook receiver helper** (`verifyWebhookSignature`/`verifyAndParseWebhook`, `createWebhookMiddleware`) ‚Äî matches the server's `X-Neko-Signature: sha256=<hex>` scheme exactly (verified against the server's real signing code, not just the docs), using `crypto.timingSafeEqual` rather than a plain string comparison.
- [x] **Migrated a real existing Neko\* project**: `API/V5/V5_UNIFIED` (found via a dedicated audit of ~30 sibling projects ‚Äî the only small, cleanly-modularized, single-provider `passport-discord` app; two other candidates were real but had auth code entangled in 2,300+/5,200+ line monolith files, not a good first migration). The direct `passport-discord` strategy was removed entirely; the SDK's OIDC result is handed to the app's *existing* Passport session via `req.login()`, so `ensureAuthenticated`/`ensureAdmin`/`req.user` and everything downstream of them (dashboard, admin) needed zero changes ‚Äî only the front door was swapped. Fixed by mounting an empty placeholder `express.Router()` synchronously in the correct position and populating it once the dynamic `import()` resolves.
  - **Verified end to end against a real, live NekoOAuth2Server instance** ‚Äî not asserted, actually run: booted the real server (`apps/server`, real Postgres-speaking scratch DB, real generated JWKS) and the real V5_UNIFIED app as two live processes, registered V5_UNIFIED as a real `Client`, and drove the full flow with `curl` and a shared cookie jar exactly the way a browser would: unauthenticated `/dashboard` ‚Üí redirected to `/login` ‚Üí `/auth/neko` ‚Üí real authorize/login/consent round trip on the real server ‚Üí code delivered back to V5_UNIFIED's real `/auth/neko/callback` ‚Üí real server-to-server token exchange + id_token verification + userinfo fetch ‚Üí a real `users` row created in V5_UNIFIED's own SQLite (confirmed via direct query: `discordId` holding the NekoOAuth2Server `sub`, correct email, `mustChangePassword: false`) ‚Üí `req.login()` into the existing Passport session ‚Üí `/dashboard` now returns 200 with real content.

## Open Questions

- Should this server also issue API keys for pure machine-to-machine Neko\* integrations (e.g. bot-to-bot), or is that out of scope and left to each project?
